Sales Story

先讲客户经营问题
再讲两类 Agent 的分工

客户通常已经有报表和数据平台，但还缺两件事：日常问题能随时问、关键分析能讲得住并推得动。

Opening Script

ChatBI Agent

解决“日常经营问数”：业务用自然语言问，系统按统一口径取数、解释和追问。

  • 类比数字 BI 工程师
  • 高频、重复、长尾问数
  • 重点讲准确率、词汇表、口径和交付闭环

Data Agent

解决“正式经营分析”：从问题拆解、证据链到结构化报告和行动建议。

  • 类比专业商业分析师
  • 目标差距、归因、复盘、专项优化
  • 重点讲证据、可核验、能过会和落动作
建议开场：我们不是替代你现有 BI 和数据平台，而是在它们之上配两类 Agent。ChatBI 让每个岗位能问数，Data Agent 让关键分析能成文、能核验、能推进。
Product Relationship

两个产品不是替代关系
而是从问数到分析交付的连续能力

ChatBI 把“问得到、取得准”做扎实；Data Agent 把“分析得透、推得动”做闭环。两者共享企业数据资产、口径、权限和业务语义。

既有数据平台数据源、报表、指标、权限边界
共享语义资产词汇表、口径、SOP、业务经验
ChatBI自然语言问数、取数解释、连续追问
Data Agent问题拆解、证据链、报告和建议
组织动作过会、复盘、转发、责任人跟进
销售注意：客户如果担心“你们是不是重建 BI”，回答要立刻拉回“基于现有 BI 和数据平台，不绕过治理”。
ChatBI Talk Track

ChatBI 的一句话
日常经营问数的数字 BI 工程师

管理层临时追问、业务岗位日常判断、经营复盘中的长尾取数，不再都排队等 BI 工程师临时查。

Digital BI Engineer
用户视角

每个岗位都能随时问

传统报表没有覆盖的临时问题，可以用业务语言直接问，不需要记表名、字段名和 SQL。

组织视角

问数变成组织资产

高频问题、业务黑话、指标口径和同义词不断沉淀，越用越稳。

人效视角

BI 团队少做重复取数

把高频、重复、长尾问数交给 Agent，BI 团队回到建模、治理和高价值分析。

建议说法：ChatBI 不是“能聊天的报表”，而是把 BI 工程师的取数、口径理解和结果解释能力，放到业务每天都会发生的问数场景里。

Data Agent Talk Track

Data Agent 的一句话
懂业务的专业商业分析师

它不是又一个聊天入口，而是把目标差距、异常归因、经营复盘和专项优化，做成可以核验、汇报和推进的正式分析交付。

Business Analyst Agent

用户价值

把问题讲圆、证据拉齐、材料成文、动作接上。业务负责人不用只靠经验点报表。

组织价值

沉淀优秀经营经验，让总部、区域、门店、渠道都能复用同一套分析方法。

人效价值

专业分析师成本高、数量少，Agent 让关键场景都能配上分析能力。

边界话术：我们不是替客户拍板，也不是承诺永远更懂业务；我们把商业分析、统计学方法和客户业务经验结合起来，让分析更容易被复核和迭代。
Buyer Reality

Data Agent 的痛点
要还原成“谁在做分析”

很多企业不是没有报表，而是关键经营分析落在了没有专业分析能力的位置上。

业务负责人自己点报表

看到指标下滑后自己下钻几层，凭经验判断原因。会上被问“为什么不是别的原因”时，讲不圆。

总部有 SOP，一线执行不出来

总部要求门店或区域写月度分析和下月计划，但真正能写好的人很少，很多报告只是照着模板填。

有经验，但不可复制

老领导、资深运营知道怎么分析，但路径来源说不清。扩门店、换团队、换区域后能力断层。

建议问客户：你们现在关键经营分析是谁在做？业务负责人自己做，运营做，还是有专门分析师？这些分析最后会不会被追问口径、路径和动作？
Pain to Proof

讲 Data Agent 时
每个功能都要闭环到一个痛点

客户买的不是截图，而是“这份分析能不能过会、转发、复核、落动作”。

分析站不住

展示任务规划、维度下钻、事实与候选原因分层。

推进链路慢

展示问题收敛成任务，直接输出报告和建议。

共识形成难

展示口径、原始数据、计算过程和证据包。

动作接不上

展示责任角色、优先动作、复盘节奏和衡量方式。

案例讲法固定顺序：客户痛点是什么，Agent 做了什么，证据在哪里，最终交付物是什么，下一步怎么让客户验证。
Demo Script

酒店月度经营分析案例
建议按三步讲

不要从“系统页面有什么按钮”开始，要从客户的经营会场景开始。

场景每月经营会前，店长要讲清目标完成率、差距来源和下月动作。
任务把“RevPAR 为什么没达标”收敛成可执行、可验收的分析任务。
证据目标、实际、完成率、渠道、房型、价格带逐层展开。
报告形成结论、证据链、原始数据入口和结构化材料。
动作输出优先动作、责任角色、跟进节奏和复盘指标。
真实案例优先。不要现场脑补没做过的行业案例，客户一旦追问行业细节，销售容易卡住。
Scheduled Delivery

别只讲“业务来问”
还要讲周期报告和强触达

对门店月报、区域周报、渠道复盘、目标完成率追踪这类固定需求，Data Agent 可以进入经营节奏，而不只是一次性问答。

固定问题目标差距、月度复盘、门店计划、渠道分析
定时触发按周、按月、按经营会节奏执行
生成报告结论、证据、原始数据、行动建议
强触达推送给店长、区域、业务负责人
复盘迭代反馈进入 SOP 和下次分析
建议说法：报表发出去不代表有人分析。Data Agent 的价值是把关键经营分析变成组织节奏，到点生成、到人触达、结果可复盘。
Objection Handling

客户常问的问题
先用统一口径接住

不要现场自由发挥。先把边界说稳，再引导客户做小范围并跑验证。

Q：准确率怎么保证？

不承诺永远正确，承诺可验证、可纠正、可迭代。

关键数值由 SQL / Python 产生，结论分层表达。PoC 用同一数据并跑验证。

Q：大模型会不会幻觉？

会，所以关键数据不交给模型口算。

模型负责规划和表达，关键计算走受控工具，证据不足只列候选因素。

Q：你怎么比客户更懂业务？

不是替客户拍板，而是把专业方法和客户经验结合。

客户提供业务规则和最终判断，系统提供分析框架、证据链和过程留痕。

Q：为什么不直接招商业分析师？

招人适合核心岗位，Agent 适合规模化覆盖场景。

专业分析师成本高、数量少，Agent 更适合覆盖门店、区域、渠道、品类等大量场景。

Competitive Talk Track

竞品不要只比功能
要比企业级交付闭环

客户可能知道市场上也有类似产品。销售要能回答：我们好在哪里，为什么值得买，ROI 怎么算。

不只比聊天入口

聊天入口只是入口。真正价值在数据接入、权限、受控执行、证据留痕、报告和行动方案。

ROI 按人效讲

ChatBI 对标数字 BI 工程师；Data Agent 对标专业商业分析师。核心是把专业能力覆盖到更多岗位。

价格不要打崩

客户预算不足时，优先用版本、范围、场景和服务边界调整，而不是随意打折。

内部提醒：竞品和报价需要持续维护 FAQ。客户知道某个厂商时，销售至少要能回答“我们比的是哪一段能力”。
Private Deployment

私有化不是只卖软件
模型、硬件和服务边界要说清

客户进入采购阶段后，关心的是数据是否出域、模型部署方式、硬件资源、实施服务和后续运营责任。

产品部署

应用、数据连接、权限、配置、基础验证。通常作为软件交付和实施服务的一部分说明。

模型部署

如果客户要求私有化模型，需要单独评估模型服务、GPU/一体机、推理成本和运维责任。

SaaS 还是私有化

SaaS 更快，私有化主要服务数据不出域、内网、审计、集团权限和本地集成要求。

建议说法：主线先讲业务价值。到部署方案阶段，再拆产品部署、模型部署、硬件资源、培训带教和持续运营。
Next Step

客户感兴趣后
不要泛泛约 Demo，要锁一个强场景

PoC 最怕“感觉不错但无法验收”。最稳妥的做法是把问题、数据、口径和验收项说具体。

下一步要拿到的信息

  • 一个具体经营问题
  • 时间范围、对象范围、核心 KPI
  • 常看维度和参考报表
  • 指标口径、权限和数据可用范围

PoC 验收看五件事

  • 核心 KPI 是否对齐
  • 一级拆解是否可复算
  • 追问能否回到原证据链
  • 边界纪律是否守住
  • 能否形成正式报告和行动建议
收尾话术：我们建议不先做大范围，而是先选一个客户最熟悉、最痛、最容易验证的数据分析场景，并跑一次，把价值闭环跑出来。
